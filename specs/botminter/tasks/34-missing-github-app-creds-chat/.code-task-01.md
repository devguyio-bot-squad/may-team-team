# Task 01: Inject GitHub App credentials into chat and meeting sessions

## Context
- **Story**: #34 — bm chat and bm meetings do not inject GitHub App credentials into the session
- **Requirements**: CRED-01, CRED-02
- **Acceptance Criteria**: AC-01, AC-02, AC-03, AC-04, AC-05, AC-06
- **Design Doc**: N/A (standalone bug fix)

## Objective
Modify `launch_session()` in `crates/bm/src/chat/mod.rs` to detect and inject GitHub App credentials (`GH_CONFIG_DIR`) into interactive chat and meeting sessions, and unset conflicting env vars (`GH_TOKEN`, `GITHUB_TOKEN`) so the member's App identity is used instead of the operator's personal auth. Emit operator-visible feedback indicating which identity is in use.

## Background
`bm start` injects GitHub App credentials via `formation/launch.rs` — both `launch_ralph()` and `launch_brain()` call `collect_bridge_env_vars()` which sets `GH_CONFIG_DIR`, and `vars_to_unset()` which removes `GH_TOKEN`/`GITHUB_TOKEN`. But `bm chat` and `bm meetings` both call `chat::launch_session()`, which launches the coding agent without any credential injection. This causes GitHub operations in interactive sessions to be attributed to the operator instead of the team member's App identity.

The credential path is `workspace/.config/gh/` — this directory is created by `resolve_app_credentials_and_deliver()` in `formation/start_members.rs` when the daemon sets up App credentials during `bm start`. The `hosts.yml` file within it contains the App's installation token.

## Technical Requirements
1. Detect if the member workspace has valid App credentials: check that `session.ws_path.join(".config/gh/hosts.yml")` exists (not just the directory — an empty or corrupted `.config/gh/` directory should not trigger injection)
2. If valid credentials exist, set `GH_CONFIG_DIR` env var on the current process before launching the coding agent
3. Unset `GH_TOKEN` and `GITHUB_TOKEN` env vars to prevent them from overriding the App credentials
4. Both code paths in `launch_session()` must inherit these env changes: the formation `exec_in` path (spawns a child process that inherits parent env) and the direct `Command::exec()` path (replaces the current process, inheriting env)
5. Emit operator-visible feedback (`eprintln!`) indicating which GitHub identity is in use: App identity with path when injected, or a note to run `bm start` when no App credentials are found

## Dependencies
- No new dependencies required
- Relies on existing credential setup from `bm start` / daemon having run at least once for the member

## Implementation Approach
1. Extract credential injection into a standalone function `inject_app_credentials(ws_path: &Path) -> bool` that:
   - Checks if `ws_path.join(".config/gh/hosts.yml")` exists
   - If yes: sets `GH_CONFIG_DIR` env var, removes `GH_TOKEN`/`GITHUB_TOKEN`, emits `eprintln!("Using GitHub App identity (GH_CONFIG_DIR: {})", ...)`, returns `true`
   - If `.config/gh/` directory exists but `hosts.yml` is missing: emits `eprintln!("Warning: App credential directory found but hosts.yml is missing. Using personal GitHub auth.")`, returns `false`
   - If `.config/gh/` doesn't exist: emits `eprintln!("Note: No App credentials found. Using personal GitHub auth. Run 'bm start' first to provision App credentials.")`, returns `false`
2. Call `inject_app_credentials(&session.ws_path)` in `launch_session()` after resolving the coding agent but before either execution path
3. This approach works because:
   - The formation `exec_in` path uses `Command::status()` which spawns a child that inherits the parent's env
   - The direct exec path uses `Command::exec()` which replaces the current process, inheriting the current env
   - `bm chat` and `bm meetings` are one-shot commands, so env pollution is not a concern
   - Note: `std::env::set_var`/`remove_var` are not thread-safe but acceptable here because `launch_session` runs on the main thread of a one-shot CLI command. This matches existing patterns in `formation/launch.rs`.
   - Known coupling: this approach depends on `exec_in` inheriting the parent process environment without sanitization. If `exec_in` is changed to use explicit env maps (e.g., for container formations per ADR-0008), credential injection must move to the `exec_in` call site.
4. Unit test the extracted `inject_app_credentials()` function directly: create temp directories with/without `.config/gh/hosts.yml` and assert env var state after each call
5. AC-04 ("both code paths covered") is a structural guarantee verified by code placement (injection before branching), not a separately testable criterion — `Command::exec()` inherits env vars by POSIX contract

## Acceptance Criteria

1. **App credentials injected when available**
   - Given a workspace with `.config/gh/hosts.yml` present (from a prior `bm start`)
   - When `launch_session()` is called (via `bm chat` or `bm meetings`)
   - Then `GH_CONFIG_DIR` env var is set to the workspace's `.config/gh/` path before the coding agent launches

2. **Conflicting tokens removed**
   - Given `GH_TOKEN` and/or `GITHUB_TOKEN` env vars are set in the operator's environment
   - When `launch_session()` detects App credentials and injects `GH_CONFIG_DIR`
   - Then `GH_TOKEN` and `GITHUB_TOKEN` are removed from the environment so they don't override `hosts.yml`

3. **No injection when credentials absent**
   - Given a workspace without `.config/gh/` directory (member has no App credentials)
   - When `launch_session()` is called
   - Then no `GH_CONFIG_DIR` is set and `GH_TOKEN`/`GITHUB_TOKEN` are left unchanged (operator's personal auth is used as fallback)

4. **Both code paths covered**
   - Given the credential injection logic runs before the execution branching
   - When `launch_session()` takes the formation `exec_in` path OR the direct `Command::exec()` path
   - Then the coding agent inherits the injected credentials in both cases (verified by code placement, not separate test — POSIX contract guarantees env inheritance)

5. **Operator feedback emitted**
   - Given App credentials are found (or not found)
   - When `launch_session()` runs
   - Then a message is emitted to stderr indicating which GitHub identity will be used (App identity with path, or personal auth fallback with guidance to run `bm start`)

6. **Corrupt credential directory handled gracefully**
   - Given `.config/gh/` directory exists but `hosts.yml` is missing or empty
   - When `launch_session()` is called
   - Then `GH_CONFIG_DIR` is NOT set, a warning is emitted about the missing `hosts.yml`, and the session falls back to personal auth
