# Code Task 02 — Make sync failure non-fatal in multi-workspace loop

## Objective

Wrap the `sync_workspace()` call in the multi-workspace loop so that a failure (including push failure) for one workspace does not abort processing of remaining workspaces. Currently, the `?` operator at `team_sync.rs:133` propagates any error from `sync_workspace()` and terminates the entire loop. This is Bug 3 from issue #39.

## Traceability

| Type | IDs |
|------|-----|
| Acceptance Criteria | AC-05, AC-06, AC-07, AC-08 |

## Scope

### Files to modify

**`crates/bm/src/workspace/team_sync.rs`:**
- Add `WorkspaceSyncFailed { name: String, error: String }` variant to `TeamSyncEvent` enum (after `WorkspaceCreateFailed`)
- Replace the `?` error propagation on `sync_workspace()` call (lines 124-133) with a match block:
  ```rust
  // Before:
  let sync_result = workspace::sync_workspace(...)?;
  events.push(TeamSyncEvent::WorkspaceSynced { ... });
  updated += 1;

  // After:
  match workspace::sync_workspace(...) {
      Ok(sync_result) => {
          events.push(TeamSyncEvent::WorkspaceSynced {
              name: member_dir_name.clone(),
              events: sync_result.events,
          });
          updated += 1;
      }
      Err(e) => {
          events.push(TeamSyncEvent::WorkspaceSyncFailed {
              name: member_dir_name.clone(),
              error: format!("{:#}", e),
          });
          failures.push(member_dir_name.clone());
          continue;
      }
  }
  ```
  This matches the existing pattern used for `create_workspace_repo` (lines 154-167).

**`crates/bm/src/commands/teams/sync.rs`:**
- Add display handler for the new `TeamSyncEvent::WorkspaceSyncFailed` variant. Format:
  ```
  ⚠ Failed to sync workspace '{name}': {error}
  ```
  Display unconditionally (not gated on verbose) since sync failures are always relevant.

### Test additions

**`crates/bm/src/workspace/team_sync.rs` or integration tests:**
- `sync_all_continues_after_workspace_failure` — set up two workspaces, sabotage one (e.g., delete the `.botminter.workspace` marker or corrupt the git state so `sync_workspace` fails), run `sync_team_workspaces`, assert the other workspace was still synced and the failed one appears in `failures`.
- `sync_all_push_failure_non_fatal` — set up two workspaces with `push: true`, configure one workspace's remote to be unreachable (point remote at a non-existent path or read-only location), run sync, assert the other workspace was still synced.

**Failure injection:** Push failure is injected by pointing one workspace's git remote at a non-existent or inaccessible path (e.g., `git remote set-url origin /nonexistent/path`). This causes `git push` to fail without needing mocks or trait abstractions. Workspace creation failure can be injected by corrupting the `.botminter.workspace` marker presence or git state.

### Design rationale

The `create_workspace_repo` path already uses `match` + `continue` for resilience (lines 154-167). The sync path should follow the same pattern. The `TeamSyncEvent::WorkspaceSyncFailed` variant provides structured reporting to the display layer, matching `WorkspaceCreateFailed`.

## Acceptance Criteria

- AC-05: Given two workspaces where the first has a push failure (no remote configured), when `sync_team_workspaces` runs with `repos: true`, then the second workspace is still synced and the first appears in the `failures` list.
- AC-06: Given a workspace where `sync_workspace` fails for any reason, when the multi-workspace loop continues, then a `TeamSyncEvent::WorkspaceSyncFailed` event is emitted with the member name and error message.
- AC-07: The display layer shows sync failures unconditionally (not gated on `--verbose`), using the format `⚠ Failed to sync workspace '{name}': {error}`.
- AC-08: `cargo test -p bm` passes. Existing sync tests are not regressed. The new resilience tests pass.

## Complexity

Low — the code change is a straightforward `?` → `match` transformation following an existing pattern in the same function. One new enum variant and its display handler. Two new tests.
