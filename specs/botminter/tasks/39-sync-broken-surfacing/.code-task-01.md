# Code Task 01 — Replace mtime comparison with content comparison in copy_if_newer

## Objective

Replace the mtime-based file freshness check in `copy_if_newer_verbose()` with a byte-content comparison. Git submodule operations produce unreliable mtimes (checkout timestamps reflect the operation time, not the file's logical age), causing `bm teams sync` to skip files that have new content but older mtimes. This is Bug 2 from issue #39.

## Scope

### Files to modify

**`crates/bm/src/workspace/util.rs`:**
- Rename `copy_if_newer_verbose()` (lines 141-161) to `copy_if_changed_verbose()` to reflect the new semantics
- Replace the mtime comparison logic (lines 145-149) with byte-content comparison:
  ```rust
  // Before:
  let src_mod = fs::metadata(src)?.modified()?;
  let dst_mod = fs::metadata(dst)?.modified()?;
  src_mod > dst_mod

  // After:
  fs::read(src)? != fs::read(dst)?
  ```

### Design rationale

Content comparison via `fs::read()` is appropriate here because:
1. The files being synced (ralph.yml, CLAUDE.md, PROMPT.md, settings.local.json) are small configuration files — typically under 50KB
2. Reading two small files is negligible overhead compared to the git operations in the same sync cycle
3. No external crate needed (no blake3/sha256) — simple byte equality is sufficient and deterministic
4. This eliminates the entire class of mtime-related bugs in git submodule workflows

Optimization: compare file lengths first (`fs::metadata(src)?.len() != fs::metadata(dst)?.len()`) as a fast-path before reading content. Most content changes also change length, making this an efficient short-circuit.

## Acceptance Criteria

- AC-01: Given source and destination files with identical content but source has an older mtime than destination, when `copy_if_changed_verbose()` is called, then the file is NOT copied (returns `Ok(false)`).
- AC-02: Given source and destination files with different content but source has an older mtime than destination, when `copy_if_changed_verbose()` is called, then the file IS copied (returns `Ok(true)` and destination has source's content). This is the critical mtime bug scenario.
- AC-03: Given `bm teams sync` is run after a git submodule update that changed a file's content but gave it an older mtime, when the sync completes, then the workspace copy has the new content.
- AC-04: `cargo test -p bm` passes. All renamed tests pass. The `filetime` crate usage can be removed from `util.rs` and `sync.rs` test modules if no longer needed.
