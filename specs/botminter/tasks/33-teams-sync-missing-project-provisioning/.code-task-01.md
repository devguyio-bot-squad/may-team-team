# Task 01: Provision new project submodules during workspace sync

## Context
- **Bug**: #33 — bm teams sync does not provision new projects into member workspaces
- **Requirements**: SYNC-01, SYNC-02, SYNC-03, SYNC-04
- **Acceptance Criteria**: AC-01, AC-02, AC-03, AC-04, AC-05, AC-06, AC-07, AC-08
- **Design Doc**: N/A (standalone bug fix)

## Objective
Modify `sync_workspace()` in `crates/bm/src/workspace/sync.rs` to detect projects present in the manifest but missing from the workspace's `projects/` directory, and provision them as git submodules with the member branch checked out. Update `team_sync.rs` to pass the manifest's project list to the sync function.

## Background
`bm teams sync` orchestrates workspace reconciliation via `sync_team_workspaces()` in `team_sync.rs`. For new workspaces (no `.botminter.workspace` marker), it calls `create_workspace_repo()` which receives `params.projects` (built from `manifest.projects` at line 91-96) and provisions all project submodules. For existing workspaces, it calls `sync_workspace()` which only updates project submodules already on disk (lines 110-131 in `sync.rs`).

The gap: `sync_workspace()` discovers projects from the filesystem (lines 176-189) and never compares against the manifest. When a new project is added to `botminter.yml` via `bm projects add`, the team repo is updated, but `sync_workspace()` doesn't know about the new project because it was never passed the manifest's project list.

The project list is already built in `team_sync.rs` at line 91-96 as `project_refs: Vec<(&str, &str)>`, but it's only used for `create_workspace_repo()` (line 145), not for `sync_workspace()` (lines 124-131).

## Technical Requirements
1. Add a `projects` parameter to `sync_workspace()` accepting `&[(&str, &str)]` (name, fork_url pairs) representing the manifest's full project list
2. After updating existing project submodules (lines 110-131), iterate the `projects` parameter and for each project not already present in `ws_root.join("projects")`, call `git_submodule_add()` and `checkout_member_branch()` to provision it
3. Ensure the `projects/` directory exists before adding new submodules (create if absent)
4. Include newly-provisioned projects in the `project_names` list (lines 176-189) so downstream operations (`inject_workspace_context`, `assemble_agent_dir_submodule`) pick them up
5. Update `team_sync.rs` to pass `&project_refs` to `sync_workspace()` at the call site (lines 124-131)
6. Update `workspace/mod.rs` public re-export if the function signature changes (it does — new parameter)
7. Emit `SyncEvent::ProjectProvisioned` for newly-provisioned projects (replaces `UpdatingSubmodule` for new projects — one event per action)
8. The command layer MUST render `ProjectProvisioned` events unconditionally (not only in verbose mode), providing non-verbose confirmation that new projects were provisioned — e.g., `"Provisioned new project: <name>"`
9. If provisioning fails for one project, continue attempting remaining projects. Wrap errors with `.with_context()` including the project name and fork URL, following the pattern at `repo.rs:281-288`. Collect failures and report them after all projects are attempted

## Dependencies
- No new dependencies required
- Relies on existing `git_submodule_add()` which is already idempotent (safe to call even if submodule exists)
- Relies on existing `checkout_member_branch()` which handles branch creation vs checkout

## Implementation Approach
1. **Modify `sync_workspace()` signature** — add `projects: &[(&str, &str)]` parameter after `github_repo`:
   ```rust
   pub fn sync_workspace(
       ws_root: &Path,
       member_dir_name: &str,
       coding_agent: &CodingAgentDef,
       verbose: bool,
       push: bool,
       project_number: Option<u64>,
       github_repo: Option<&str>,
       projects: &[(&str, &str)],  // NEW: (name, fork_url) from manifest
   ) -> Result<SyncResult> {
   ```

2. **Add new project provisioning** — after the existing project update loop (line 131) and before the context file re-copy (line 133), add a block that:
   - Ensures `projects/` directory exists (`fs::create_dir_all`)
   - Iterates `projects` and checks if `ws_root.join("projects").join(name)` exists
   - For each missing project: calls `git_submodule_add(ws_root, fork_url, &format!("projects/{}", name))` and `checkout_member_branch(&ws_root.join("projects").join(name), member_dir_name, verbose, &mut result)`
   - Wraps `git_submodule_add()` and `checkout_member_branch()` failures with `.with_context()` including project name and fork URL (following the pattern at `repo.rs:281-288`)
   - On failure: logs the error, continues to the next project, and collects failures for reporting
   - Emits `SyncEvent::ProjectProvisioned(String)` for each successfully provisioned project (the command layer renders this unconditionally, not only in verbose mode)

3. **Update `team_sync.rs`** — pass `&project_refs` to `sync_workspace()` at lines 124-131:
   ```rust
   let sync_result = workspace::sync_workspace(
       &ws,
       member_dir_name,
       params.coding_agent,
       params.verbose,
       params.repos,
       params.project_number,
       params.github_repo,
       &project_refs,  // NEW
   )?;
   ```

4. **Update callers** — the `sync_workspace()` function is also called from approximately 16 test call sites in `sync.rs` and 1 in `team_sync.rs`. Tests that don't exercise new-project provisioning pass `&[]` as the projects parameter. This is a mechanical change — no logic modifications.

5. **Project names reconciliation** — the `project_names` list at lines 176-189 already discovers from the filesystem, so newly-added submodules will be automatically included. No change needed there.

6. **Test the fix** — add a new test `sync_provisions_new_project` that:
   - Creates a workspace via `setup_syncable_workspace()`
   - Verifies `projects/` has no project submodules initially
   - Calls `sync_workspace()` with a project list containing one project
   - Asserts the project directory exists, is a git repo, and is on the member branch
   - Calls `sync_workspace()` again to verify idempotency (no error, no duplicate)

7. This approach works because:
   - `git_submodule_add()` is idempotent — if the submodule already exists, it returns `Ok(())` without changes
   - `checkout_member_branch()` handles both existing and new branches
   - The filesystem-based `project_names` discovery at lines 176-189 runs after the new provisioning block, so it naturally picks up newly-added projects for context injection and agent dir assembly
   - Known coupling: project fork URLs must be accessible from the workspace machine. This is the same constraint as `create_workspace_repo()`. For push-mode workspaces with GitHub remotes, the URL must be a valid Git remote.

## Acceptance Criteria

1. **New project provisioned during sync**
   - Given a workspace that was created with zero projects
   - When `bm teams sync` runs and the manifest contains a new project (added via `bm projects add`)
   - Then the workspace's `projects/<project-name>/` directory is created as a git submodule pointing to the project's fork URL

2. **Member branch checked out in new project**
   - Given a new project submodule is provisioned during sync
   - When the provisioning completes
   - Then the submodule is checked out on the member's branch (not detached HEAD)

3. **Existing projects not affected**
   - Given a workspace with existing project submodules
   - When `bm teams sync` runs with the same manifest (no new projects)
   - Then existing project submodules are updated normally (fetch + merge) with no errors or regressions

4. **Idempotent provisioning**
   - Given a workspace where a new project was already provisioned in a prior sync
   - When `bm teams sync` runs again with the same manifest
   - Then no duplicate submodule is created, no error occurs, and the workspace remains valid

5. **Agent dir and context updated for new project**
   - Given a new project submodule is provisioned during sync
   - When context injection and agent dir assembly run (later in the sync flow)
   - Then the new project is included in the workspace's context (CLAUDE.md) and agent directory symlinks

6. **Non-verbose feedback for new projects**
   - Given any run mode (verbose or not)
   - When a new project submodule is provisioned
   - Then a `SyncEvent::ProjectProvisioned` event is emitted and the command layer renders it unconditionally (e.g., `"Provisioned new project: <name>"`)

7. **Graceful handling of provisioning failure**
   - Given a manifest with two new projects where the second has an unreachable fork URL
   - When `sync_workspace()` runs
   - Then the first project is provisioned successfully, the second fails with an actionable error message including the project name and fork URL, and the function reports the failure without aborting the entire sync

8. **Actionable error message for failed provisioning**
   - Given a project with an invalid or unreachable fork URL
   - When `git_submodule_add()` fails during provisioning
   - Then the error includes the project name and fork URL (not a raw git stderr dump), following the `.with_context()` pattern from `repo.rs:281-288`
