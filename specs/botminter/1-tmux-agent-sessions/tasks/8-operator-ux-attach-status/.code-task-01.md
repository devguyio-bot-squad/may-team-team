# Task 01: Add tmux prerequisite check to `check_prerequisites()`

## Context
- **Story**: [#8 — STORY-04: Add operator UX — bm attach, bm status tmux info, prerequisites, E2E tests](https://github.com/devguyio-bot-squad/may-team-team/issues/8)
- **Requirements**: TMUX-02
- **Acceptance Criteria**: AC-01
- **Design Doc**: team/specs/botminter/1-tmux-agent-sessions/design.md

## Objective
Add tmux availability and version checking to `LinuxLocalFormation::check_prerequisites()` so that `bm start` fails early with actionable error messages when tmux is missing or too old.

## Background
`check_prerequisites()` in `crates/bm/src/formation/local/linux/mod.rs` (lines 76-80) currently only checks for `ralph` in PATH. The design requires tmux 3.0+ as a hard prerequisite for the local formation. `TmuxSession::check_tmux_available()` (implemented in STORY-01, `formation/local/tmux/`) provides version detection and parsing — this task wires it into the prerequisites check that runs at `bm start` time.

There is also a `check_environment()` method (lines 47-73) that checks `ralph` and `just`. The tmux check belongs in `check_prerequisites()` since tmux is a runtime dependency (like `ralph`), not an environment configuration concern.

## Technical Requirements
1. Call `TmuxSession::check_tmux_available()` inside `LinuxLocalFormation::check_prerequisites()`
2. On tmux not found: error message MUST name "tmux" and suggest installation commands for at least two package managers (e.g., `apt install tmux` / `dnf install tmux`)
3. On tmux version < 3.0: error message MUST show the found version number and state that 3.0+ is required
4. The check MUST run before any tmux session operations — `check_prerequisites()` is called at the top of the start flow

## Dependencies
- `TmuxSession::check_tmux_available()` from STORY-01 (Task 02 of story #5) must be implemented
- `formation/local/tmux/mod.rs` module must exist and be registered

## Implementation Approach
1. Add `use super::tmux::TmuxSession;` (or appropriate path) to `formation/local/linux/mod.rs`
2. In `check_prerequisites()`, after the existing `ralph` check, call `TmuxSession::check_tmux_available()`
3. Map the error result to user-friendly messages matching the design's error handling table:
   - Not found: "tmux is required but not found. Install it with: apt install tmux / dnf install tmux"
   - Version too old: "tmux 3.0+ is required (found X.Y). Please upgrade."
4. The error messages should match the exact wording in the design doc for consistency with AC-01

## Acceptance Criteria

1. **Missing tmux detection**
   - Given tmux is not installed (not in PATH)
   - When the operator runs `bm start`
   - Then the command exits with non-zero status, the error names "tmux", and suggests installation commands for at least two package managers

2. **Old tmux version detection**
   - Given tmux is installed but version is below 3.0
   - When the operator runs `bm start`
   - Then the command exits with non-zero status, shows the found version number, and states 3.0+ is required

3. **Successful prerequisite check**
   - Given tmux 3.0+ is installed and `ralph` is in PATH
   - When `check_prerequisites()` is called
   - Then it returns `Ok(())` without error

4. **Missing tmux test**
   - Given tmux is not in PATH (e.g., test subprocess with controlled PATH excluding tmux)
   - When `check_prerequisites()` is called
   - Then the error message contains the string "tmux" and at least two package manager install commands

5. **Old version tmux test**
   - Given a mock `tmux` script in PATH that outputs a version string below 3.0 (e.g., "tmux 2.9")
   - When `check_prerequisites()` is called
   - Then the error message contains the found version number and the string "3.0+"
