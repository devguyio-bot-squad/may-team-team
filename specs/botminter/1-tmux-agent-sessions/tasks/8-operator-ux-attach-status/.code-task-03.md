# Task 03: Add tmux session info to `bm status`

## Context
- **Story**: [#8 — STORY-04: Add operator UX — bm attach, bm status tmux info, prerequisites, E2E tests](https://github.com/devguyio-bot-squad/may-team-team/issues/8)
- **Requirements**: UX-02, UX-03
- **Acceptance Criteria**: AC-11
- **Design Doc**: team/specs/botminter/1-tmux-agent-sessions/design.md

## Objective
Add tmux session information to `bm status` output, showing the session name, window count, and both `bm attach` and raw tmux attach commands.

## Background
`StatusInfo` in `crates/bm/src/state/dashboard.rs` (lines 19-28) currently has fields for formation, projects, daemon, members, bridge, and verbose info. The `gather_status()` function (lines 93-180) collects this data. `commands/status.rs` formats and displays it. None of these currently reference tmux.

The design specifies adding a `TmuxStatusInfo` struct with session name, socket name, window count, `attach_command` ("bm attach"), and `raw_attach_command` ("tmux -L botminter attach -t bm-<team>"). The display should show:
```
  tmux: bm-may-team (3 windows)
  attach: bm attach  (or: tmux -L botminter attach -t bm-may-team)
```

## Technical Requirements
1. Create `TmuxStatusInfo` struct in `state/dashboard.rs`:
   - `session_name: String`
   - `socket_name: String`
   - `window_count: usize`
   - `attach_command: String` — "bm attach"
   - `raw_attach_command: String` — "tmux -L botminter attach -t bm-<team>"
2. Add `pub tmux: Option<TmuxStatusInfo>` field to `StatusInfo`
3. In `gather_status()`, construct a `TmuxSession` from the team name, call `session_info()` if the session exists, and populate `TmuxStatusInfo`
4. In `commands/status.rs`, display tmux info after daemon info when `tmux` is `Some`:
   - Line 1: `tmux: bm-<team> (N windows)`
   - Line 2: `attach: bm attach  (or: tmux -L botminter attach -t bm-<team>)`
5. When no tmux session exists, `tmux` should be `None` and no tmux section is displayed

## Dependencies
- `TmuxSession` struct with `new()`, `exists()`, and `session_info()` methods from STORY-01/02 (must be implemented)
- `SessionInfo` struct from `formation/local/tmux/mod.rs` with `session_name`, `socket_name`, `windows` fields

## Implementation Approach
1. Add `TmuxStatusInfo` struct definition in `state/dashboard.rs`
2. Add `tmux: Option<TmuxStatusInfo>` field to `StatusInfo`
3. In `gather_status()`: after the existing member/daemon gathering, try to construct `TmuxSession::new(&team_name)` and check `tmux.exists()`. If the session exists, call `tmux.session_info()` and map to `TmuxStatusInfo`. If it doesn't exist or construction fails, set `tmux: None`.
4. In `commands/status.rs`: add a display block for `status_info.tmux` after the daemon section, matching the format from the design doc
5. Integration tests: verify `bm status` output contains tmux info when session exists, and omits it when no session exists

## Acceptance Criteria

1. **Tmux info in status output**
   - Given agents are running in tmux
   - When the operator runs `bm status`
   - Then the output includes the tmux session name (e.g., `bm-may-team`)
   - And the output includes the window count
   - And the output includes the `bm attach` command

2. **Raw attach command shown**
   - Given agents are running in tmux
   - When the operator runs `bm status`
   - Then the output includes the raw tmux attach command (e.g., `tmux -L botminter attach -t bm-may-team`)

3. **No tmux info when session absent**
   - Given no tmux session exists
   - When the operator runs `bm status`
   - Then no tmux section appears in the output

4. **Graceful degradation on tmux error**
   - Given tmux is installed but `session_info()` returns an error (e.g., corrupted socket)
   - When the operator runs `bm status`
   - Then the tmux section is omitted (treated as `None`) and no stack trace or panic is shown

5. **Integration test coverage**
   - Given the status display implementation
   - When running the test suite
   - Then tests verify tmux info presence/absence in status output based on session state
