# Task 01: Rewrite `launch_ralph()` and `launch_brain()` to use tmux windows

## Context
- **Story**: #7 — STORY-03: Wire tmux into launch functions and start/stop orchestration
- **Requirements**: TMUX-01, TMUX-04, SESS-02, SESS-03, SESS-04
- **Acceptance Criteria**: AC-04, AC-05, AC-15, AC-16
- **Design Doc**: team/specs/botminter/1-tmux-agent-sessions/design.md

## Objective
Replace bare background process spawning in `launch_ralph()` and `launch_brain()` with tmux window creation via `TmuxSession::create_window()`. After this task, both launch functions accept a `&TmuxSession` parameter and create named tmux windows instead of detached processes. The daemon remains a background process — only agent members get tmux windows.

## Background
Currently `launch_ralph()` spawns `ralph run -p PROMPT.md` as a detached process with `Stdio::null()` on all three channels and calls `reap_child()` to prevent zombies. `launch_brain()` does the same but tees stderr to `brain-stderr.log`. Both return the child PID.

With tmux, the process runs inside a tmux pane — stdout and stderr go directly to the pane (visible to the operator). The PID comes from `#{pane_pid}` instead of `child.id()`. `reap_child()` is no longer needed for member launches because tmux manages the process. Environment variables (credentials, bridge tokens) are passed via `-e` flags for security.

## Technical Requirements
1. Add `tmux: &TmuxSession` parameter to `launch_ralph()` signature
2. Add `tmux: &TmuxSession` parameter to `launch_brain()` — either as a field on `BrainLaunchConfig` or as a separate parameter (match the design doc's approach)
3. In both functions: build the command as `&[&str]` (e.g., `&["ralph", "run", "-p", "PROMPT.md"]`), NOT a shell string
4. Collect env vars as `Vec<(&str, &str)>` from the existing bridge/credential logic
5. Call `tmux.create_window(member_name, cmd, workspace, &envs)` instead of `Command::new().spawn()`
6. Remove `Stdio::null()` redirections — tmux pane captures output natively (satisfies SESS-03)
7. Remove `reap_child()` calls for member launches — tmux manages the process. Retain `reap_child()` for daemon spawning.
8. Brain stderr: the command inside the tmux window must tee stderr to `brain-stderr.log` while still showing it in the pane. The preferred approach is to have the brain binary itself tee stderr to a file internally (it controls its own startup), avoiding any shell wrapper. If process substitution (`2> >(tee ...)`) is used instead, it requires `bash -c "..."` wrapping which contradicts requirement #3 — document that this specific window uses a shell invocation, and verify that no credential values appear in the shell string (credentials go via `-e` flags, not interpolation).
9. The returned PID (from `#{pane_pid}` via `create_window()`) is recorded in state.json as before
10. Update `formation/mod.rs` re-exports to include the new `TmuxSession` parameter in `launch_ralph` and `launch_brain` signatures
11. The daemon PID MUST NOT appear in any tmux window (SESS-04)

## Implementation Approach
1. Modify `launch_ralph()`:
   - Add `tmux: &TmuxSession` as first parameter
   - Build cmd: `&["ralph", "run", "-p", "PROMPT.md"]`
   - Collect env vars: iterate the existing `cmd.env()` calls and collect them into a `Vec<(&str, &str)>` instead
   - Replace `cmd.spawn()` + `reap_child()` with `tmux.create_window(member_name, &cmd_args, workspace, &envs)?`
   - Return the PID from `create_window()`
   - Note: `member_name` is not currently a parameter — derive it from the workspace path or add it as a parameter
2. Modify `launch_brain()`:
   - Add `tmux: &TmuxSession` to `BrainLaunchConfig` or as a separate parameter
   - Build cmd: the current `bm brain-run --workspace ... --system-prompt ...` args
   - Brain stderr tee: wrap the command in a shell invocation that tees stderr, OR change the brain binary to log stderr to a file internally while still writing to stderr
   - Replace spawn + reap with `tmux.create_window()`
3. Update `formation/mod.rs` re-exports
4. Write integration tests:
   - `launch_ralph(&tmux, ...)` with a stub command → window exists with correct name → PID matches running process
   - `launch_brain(&tmux, ...)` → verify `brain-stderr.log` is written alongside tmux pane output
   - Env vars passed via `-e` are NOT visible in `#{pane_start_command}` (primary assertion) or `ps -p <pid> -o args=` (secondary assertion)
   - Verify daemon PID is NOT in any tmux window

## Acceptance Criteria

1. **Ralph Launch Creates Window**
   - Given a tmux session exists
   - When `launch_ralph(&tmux, workspace, ...)` is called
   - Then a tmux window exists named after the member, and the returned PID matches a running process inside the pane

2. **Brain Launch Creates Window with Stderr Log**
   - Given a tmux session exists
   - When `launch_brain(&tmux, &config)` is called with a stub command that writes a known string (e.g., `"BRAIN_STDERR_TEST"`) to stderr
   - Then a tmux window exists, `brain-stderr.log` contains `"BRAIN_STDERR_TEST"`, and `tmux capture-pane` output also contains `"BRAIN_STDERR_TEST"`

3. **Agent Output Visible in Pane**
   - Given agents are running in tmux
   - When `tmux capture-pane -t <session>:<member> -p` is polled (every 500ms, up to 10s)
   - Then the captured output contains agent process output (stdout or stderr)
   - Test approach: use a polling loop with backoff, not a single capture after a fixed delay, to avoid CI flakiness under load.

4. **Daemon Not in Tmux**
   - Given agents are running in tmux
   - When `tmux list-windows` is checked
   - Then no window contains the daemon process name or PID

5. **Credential Security**
   - Given agents are launched with env vars (tokens, credentials)
   - When `#{pane_start_command}` and `ps -p <pid> -o args=` are inspected
   - Then no credential values appear in the output

6. **TMUX_TMPDIR Isolation (regression guard)**
   - Given `TMUX_TMPDIR` is set in the environment
   - When agents are launched
   - Then the tmux socket is under `/tmp/tmux-<uid>/`, not under `TMUX_TMPDIR`
   - Note: the behavior is implemented in STORY-01/02's `tmux_cmd()` helper. This AC is a regression guard verifying it holds through the launch integration.
